# Your First 60 Minutes in AWS

*What everyone skips and regrets later*

## Why This Matters

Picture this: You create an AWS account, generate root access keys, and share them on Slack so everyone can deploy. You're moving fast.

Three months later, you wake up to a $47,000 bill. Cryptominers have been running on your account for weeks. The worst part? CloudTrail wasn't enabled, so you can't even see how they got in.

This happens constantly. Not because engineers are careless, but because AWS security docs assume you have a compliance team and six months to implement everything. You don't. You have a product to ship.

This guide is the middle path: **enough security to sleep at night, without the enterprise overhead that kills velocity.**

AWS operates on a shared responsibility model. They secure the infrastructure‚Äîthe physical data centers, the hypervisors, the network. You secure your account, your data, and your configurations.

<Alert type="important" title="AWS Shared Responsibility Model">

**Your Responsibility:**
- Customer data & encryption
- Identity & access management (IAM)
- Application security
- Operating system & network configuration
- Firewall & security group rules

**AWS Responsibility:**
- Physical security of data centers
- Hardware & infrastructure
- Network infrastructure
- Virtualization layer
- Managed service security (S3, RDS, Lambda, etc.)

</Alert>

## Root Account Lockdown

### What Root Is (And Why It's Dangerous)

When you create an AWS account, you get a single identity: the root user. This account has **unlimited privileges**. It can:

- Bypass every IAM policy you create
- Delete CloudTrail logs (hiding evidence of breaches)
- Close the entire AWS account
- Do anything, to any resource, with no restrictions

There is no way to limit root's power. That's by design‚Äîit's the "break glass" account for emergencies. But that same unlimited power makes it the single most dangerous credential in your infrastructure.

**The goal: Use root once to set up your account, then never again.**

<Alert type="warning" title="The Fine Line">

‚ùå **Under-engineering:** Using root for daily work, sharing root credentials

‚úÖ **Right-sized:** Root locked with MFA, used only for root-only tasks

‚ùå **Over-engineering:** Hardware security keys in bank vaults, 3-person approval workflows (save this for when you're at scale)

</Alert>

### Delete Root Access Keys

Root access keys are long-lived credentials that grant unlimited API access. If they exist, delete them immediately.

<Step number={1} description="Delete Root Access Keys">
  1. Sign in to the [AWS Console](https://console.aws.amazon.com/) as root
  2. Click your account name in the top-right corner
  3. Select **Security credentials**
  4. In the **Access keys** section, check if any keys exist
  5. If keys exist, click **Actions** ‚Üí **Delete**
</Step>

<Alert type="caution">

If you're using these keys somewhere, you have a bigger problem. Stop, figure out where they're used, and replace them with IAM role credentials. We'll cover proper credential management in Part 3.

</Alert>

### Enable MFA on Root

MFA adds a second authentication factor. Even if someone steals your password, they can't sign in without physical access to your MFA device.

**Your options:**

| Type | Security | Convenience | Recommendation |
|------|----------|-------------|----------------|
| Passkey (FIDO2) | Highest | High | Best for most |
| Authenticator app | High | High | Good alternative |
| Hardware TOTP token | Highest | Low | High-security orgs only |

For most startups, a passkey or authenticator app (1Password, Authy, Google Authenticator) works fine.

**To enable MFA:** Follow the [AWS guide to enable MFA on root](https://docs.aws.amazon.com/IAM/latest/UserGuide/enable-mfa-for-root.html).

<Alert type="important">

**Critical:** Store your MFA recovery codes somewhere safe. If you lose access to your MFA device and don't have recovery codes, you'll go through AWS support hell to regain access.

</Alert>

### When You'll Actually Need Root

After setup, you'll rarely need root. But these tasks require it:

- Close your AWS account
- Change your account name or root email
- Activate IAM access to Billing (one-time setup)
- Restore IAM permissions if completely locked out
- Sign up for GovCloud
- Change your AWS Support plan
- Edit/delete certain resource policies that deny all principals (S3, SQS edge cases)

Full list: [Tasks requiring root credentials](https://docs.aws.amazon.com/IAM/latest/UserGuide/root-user-tasks.html)

For everything else, use an IAM admin user‚Äîwhich we'll create next.

## Your First IAM Admin User

### Why Not Just Use Root with MFA?

MFA protects against password theft, but it doesn't reduce root's power. A mistake made while signed in as root‚Äîdeleting the wrong resource, misconfiguring a policy‚Äîhas no safety net.

An IAM admin user has the same practical capabilities for daily work, but with key differences:

- Actions are logged in CloudTrail with the specific user identity
- You can revoke access without losing account control
- You can add restrictions later if needed
- It creates separation: root for emergencies, admin for everything else

### Create the Admin Group

Create a group first, then add users to it. This pattern scales when you add team members later.

<Step number={1} description="Create Admin Group">
  1. Sign in to the [IAM console](https://console.aws.amazon.com/iam/)
  2. In the left navigation, click **User groups**
  3. Click **Create group**
  4. For **Group name**, enter `Administrators`
  5. In the **Attach permissions policies** section, search for `AdministratorAccess`
  6. Select the checkbox next to **AdministratorAccess**
  7. Click **Create group**
</Step>

### Create Your Admin User

<Step number={2} description="Create IAM Admin User">
  1. In the IAM console, click **Users** in the left navigation
  2. Click **Create user**
  3. Enter a username (e.g., `yourname-admin`‚Äîuse your name, not a generic "admin")
  4. Select **Provide user access to the AWS Management Console**
  5. Choose **I want to create an IAM user** (not Identity Center for now‚Äîwe'll cover that in Part 2)
  6. Set a strong password
  7. Uncheck **Users must create a new password at next sign-in** (you're the only user)
  8. Click **Next**
  9. Select the **Administrators** group you created
  10. Click **Next**, then **Create user**
</Step>

### Enable MFA on Your Admin User

<Step number={3} description="Enable MFA">
  1. In the IAM console, click **Users**
  2. Click your new admin username
  3. Select the **Security credentials** tab
  4. In the **Multi-factor authentication (MFA)** section, click **Assign MFA device**
  5. Follow the prompts to set up your authenticator app or passkey
</Step>

### Switch to Your Admin User

<Step number={4} description="Switch to Admin User">
  1. Sign out of root
  2. Go to your AWS sign-in URL: `https://YOUR_ACCOUNT_ID.signin.aws.amazon.com/console`
  3. Sign in with your new IAM admin credentials
  
  **Bookmark this URL.** You'll use it daily.
</Step>

### Set an Account Alias

That 12-digit account ID is hard to remember. Create an alias:

<Step number={5} description="Set Account Alias">
  1. In the IAM console, find **AWS Account** on the right side of the dashboard
  2. Next to **Account Alias**, click **Create**
  3. Enter something memorable (e.g., `mycompany-prod`)
  4. Click **Save changes**
  
  Your new sign-in URL: `https://mycompany-prod.signin.aws.amazon.com/console`
</Step>

<Alert type="tip" title="The Fine Line">

‚ùå **Under-engineering:** Everyone shares one IAM user, no MFA

‚úÖ **Right-sized:** One admin user per person, MFA required

‚ùå **Over-engineering:** Full IAM Identity Center + SSO setup before you have paying customers (we'll cover this in Part 2)

</Alert>

## Billing Protection

AWS bills are post-paid. You find out what you owe 30 days after you've spent it. Without alerts, a misconfiguration or attack can run up thousands of dollars before you notice.

### Activate IAM Billing Access

By default, only root can view billing information. That's a problem‚Äîyou want to avoid using root. Fix this now:

<Step number={1} description="Activate IAM Billing Access">
  1. Sign in as **root** (last time for a while)
  2. Click your account name ‚Üí **Account**
  3. Scroll to **IAM user and role access to Billing information**
  4. Click **Edit**
  5. Select **Activate IAM Access**
  6. Click **Update**
  
  Now your IAM admin user can view and manage billing.
</Step>

### Create a Budget Alert

AWS Budgets notifies you before costs spiral.

<Step number={2} description="Create Budget Alert">
  1. Sign in as your IAM admin user
  2. Go to **Billing and Cost Management** ‚Üí **Budgets**
  3. Click **Create budget**
  4. Select **Customize (advanced)**
  5. Choose **Cost budget**, click **Next**
  6. Configure your budget:
     - **Name:** `Monthly spend alert`
     - **Period:** Monthly
     - **Budget amount:** Fixed, `$50` (adjust based on expected spend)
  7. Click **Next**
  8. Add alert thresholds (see table below)
  9. For each threshold, add your email address
  10. Click **Next**, then **Create budget**
</Step>

**Recommended Alert Thresholds:**

| Threshold | What It Means |
|-----------|---------------|
| 50% | Halfway through your budget‚Äînormal check-in |
| 80% | Investigate if this is unexpected |
| 100% | You've hit your limit |
| 150% | Something is probably wrong |

<Alert type="warning">

**Important:** AWS Budgets don't *stop* spending. They only notify you. If you need hard spending limits, look into AWS Organizations SCPs‚Äîwe'll cover that in Part 2.

</Alert>

<Step number={3} description="Enable Free Tier Alerts">
  1. Go to **Billing and Cost Management** ‚Üí **Billing preferences**
  2. Select **Receive Free Tier Usage Alerts**
  3. Enter your email address
  4. Click **Save preferences**

## CloudTrail: Your Audit Log

CloudTrail records every API call made in your AWS account. Who did what, when, and from where. Without it, you're flying blind.

### What CloudTrail Logs

- Console sign-ins (successful and failed)
- API calls from CLI, SDK, and console actions
- Resource changes (create, modify, delete)
- IAM permission changes

### What CloudTrail Doesn't Log by Default

- S3 object-level access (reads/writes to individual files)
- Lambda function invocations
- Data events in general

These cost extra and generate massive log volumes. For now, the default management events are enough. We'll add data events when you have something worth protecting.

### Create a Trail

<Step number={1} description="Create CloudTrail">
  1. Go to the [CloudTrail console](https://console.aws.amazon.com/cloudtrail/)
  2. Click **Create trail**
  3. Configure the trail:
     - **Trail name:** `management-events`
     - **Storage location:** Create new S3 bucket (let AWS generate the name)
     - **Log file SSE-KMS encryption:** Enabled (use AWS managed key for now)
  4. Under **Additional settings:**
     - **Log file validation:** Enabled (this detects tampering)
  5. Click **Next**
  6. For **Events:**
     - Keep **Management events** selected
     - **API activity:** Both Read and Write
  7. Click **Next**, then **Create trail**
  
  Your account is now recording everything. This log is your forensic evidence if anything ever goes wrong.
</Step>

## Quick Wins

A few more items that take 5 minutes but save headaches later.

### Region Selection

AWS has 30+ regions worldwide. For your first project, pick **one primary region** and stick with it.

**Recommendation: `us-east-1` (N. Virginia)**

Why:
- Gets new AWS features first
- Most tutorials and documentation use it as the example
- Generally cheapest for most services
- Best SDK and tooling support

**But shouldn't I pick the closest region for latency?**

For user-facing applications, yes. But you're setting up infrastructure foundations, not serving users yet. Start with `us-east-1`, expand to other regions when latency actually matters for your users.

**Test your latency to AWS regions:**

<AwsRegionLatency client:load />

**Region Selection Factors:**

| Factor | Consideration |
|--------|---------------|
| User latency | Where are your users? (matters later) |
| Compliance | GDPR requires EU regions for EU data |
| Service availability | Not all services exist in all regions |
| Pricing | Can vary 10-20% between regions |
| Feature releases | us-east-1 gets features first |

<Alert type="tip">

**Coming later:** Multi-region deployments are a future topic. Don't over-engineer until you have users complaining about latency.

</Alert>

### Tagging Strategy

Tags are key-value labels on AWS resources. They seem optional until you have 200 resources and can't tell which project they belong to or whether they're safe to delete.

**Start with these three tags on everything:**

| Tag | Example Values | Why You Need It |
|-----|----------------|-----------------|
| Environment | prod / staging / dev / sandbox | Know what's safe to delete or modify |
| Project | website / api / ml-pipeline | Track costs per project, find related resources |
| Owner | yourname / platform-team | Know who to ask about this resource |

<Alert type="note">

For now, remember to add these manually when creating resources. In Part 4, we'll enforce required tags with Terraform and catch untagged resources automatically.

</Alert>
